# Changelog

All notable changes to BearClawWeb are documented here.

## [Unreleased]

### Added

- CI: added `publish` job to `.github/workflows/ci.yml` that builds and pushes `ghcr.io/bare-systems/bear-claw-web` to GitHub Container Registry on every push to `main`; builds without pushing on PRs.

### Added

- Added a Portal single-sign-on handoff so a user authenticated at the public BareSystems Portal is signed into this device's BearClawWeb without a second login. A new `PortalIdentityToken` service (mirroring `TardigradeIdentityToken`) verifies the Portal's short-lived HS256 identity assertion against a shared `PORTAL_SSO_SECRET`, and `SessionsController#portal` (route `GET /auth/portal`, env-gated by `PORTAL_SSO_ENABLED`) establishes a session from it. Existing accounts are matched by email so a user's Google identity (`google_uid`) is never clobbered; genuinely new users still go through the invite-gated `User.from_omniauth` path. The token's per-device `role` claim is applied to this instance — `owner → operator` (edit), invited `viewer → viewer` (read-only) — and an existing admin is never demoted. Covered in `test/controllers/sessions_controller_test.rb` (owner/viewer roles, admin preservation, google_uid preservation, wrong-secret/expired/uninvited rejection, disabled-route hiding). Also added `app.baresystems.com` to `config.hosts` so BearClawWeb accepts requests arriving through the homelab's existing Cloudflare tunnel (cloudflared → `127.0.0.1:6701`), which is how the Portal hands users off.
- Added a "DNS Talkers" panel to the `Security → Network` page, backed by Ursa Major's new Pi-hole DNS insight API (`/api/v1/dns/talkers`, ROADMAP Phase 5B). It shows which devices made the most DNS queries over the last 24h with a per-client blocked/allowed split and a header summary (total queries · blocked · block ratio); clients with a high block ratio (≥25%) are badged amber. Best-effort: an older Ursa without the endpoint (404) or an unreadable Pi-hole DB degrades to an "unavailable" note without blanking the device inventory table. The Ursa bearer token stays server-side. Tested in `test/controllers/security/network_controller_test.rb`.
- Added a dedicated home-network inventory table under `Security → Network` (`Security::NetworkController`, admin-only). It lists every recorded device (IP, MAC, vendor, label, trust badge, last-seen, times-seen) with Total/Trusted/Unknown summary cards, a "Set Baseline" action that marks all currently-seen devices trusted, per-device Trust/Untrust toggles, and an inline label editor. Unknown (unbaselined) devices are row-highlighted amber. All Ursa calls are proxied server-side through `UrsaClient`, so the Ursa bearer token never reaches the browser.
- Added a home-network device-inventory dashboard widget backed by Ursa Major's new network-insight API (ROADMAP Phase 5D). `Home::UrsaDeviceSync` now reads `/api/v1/network/devices` and upserts a "Network Devices" `status` capability on the `ursa-overview` device — total device count with a Total/Trusted/Unknown breakdown. The status badge turns amber (`warning`) when any unknown/unbaselined device is present, surfacing the "a new device joined the network" security signal, and stays green (`active`) when the network matches the trusted baseline. The widget is selectable as a "Security Counter" like the existing session/campaign counters and the Ursa bearer token stays server-side. Degrades gracefully: an older Ursa deploy without the endpoint (404) logs and is skipped without aborting the session/campaign sync.
- Added Kodiak-backed source-monitoring panels to the Finances dashboard. BearClawWeb now reads Kodiak's `/api/v1/signals/overview`, `/api/v1/signals/alerts`, and `/api/v1/signals/sources` endpoints through `KodiakClient` and renders three new operator surfaces: recent deterministic buy/sell alerts, a `needs_inference` queue for posts that Kodiak could not classify without guessing, and per-source poll health/error state.
- Added first-party X OAuth connection flow under `Settings → Integrations`. BearClawWeb now starts the X OAuth 2.0 PKCE flow, handles the callback at `/settings/integrations/x/callback`, records connection metadata locally, and hands the resulting X user-context tokens to Kodiak so Kodiak can read the authenticated home timeline directly.
- Added outdoor weather and outdoor air-quality widgets sourced from Polar's NOAA and AirNow feeds. `PolarClient` gained `weather_current` (`/v1/weather/current`) and `air_quality_current` (`/v1/air-quality/current`), and `Home::PolarDeviceSync` now upserts an "Outdoor Weather" device (temperature, humidity, wind speed, pressure, condition) and an "Outdoor Air Quality" device (overall AQI, with per-pollutant AQI sub-indices kept in device metadata) as `scope: "outdoor"` sensor capabilities. These outdoor feeds are best-effort: a missing AirNow token or a stale feed no longer fails the indoor-sensor sync. `Home::MetricPresenter` gained icon/threshold bands for `aqi` (EPA categories), `wind_speed`, and `pressure`, and `Home::DashboardProvisioner` idempotently back-fills the home dashboard with the outdoor temperature, condition, wind-speed, and AQI widgets even when indoor Polar widgets already exist.
- Added an env-gated secondary OIDC "Support Login" path for BearClaw. Google OAuth remains the primary production login, but BearClaw can now expose a separate Keycloak-backed test login when `OIDC_SUPPORT_ENABLED` and the related OIDC env vars are configured. The support path reuses OmniAuth, can be restricted to an explicit `OIDC_ALLOWED_EMAILS` allowlist, and gives us a stable browser-testing login without depending on a personal Google SSO session.
- Added a "Recent Alerts" dashboard section backed by Koala's new `/admin/alerts` feed (`KoalaClient#recent_alerts`). Surfaces people/packages actually detected — sky/emerald themed, with zone · camera · confidence and relative time — replacing the old "stale = no detection" framing. A quiet feed now reads as healthy.
- Added detection-snapshot thumbnails to each "Recent Alerts" card: the operator now sees the bbox-annotated camera frame Koala captured at the moment of detection. Proxied through `Home::AlertsController#snapshot` → `KoalaClient#alert_snapshot` (`GET /admin/alerts/{id}/snapshot`) so the Koala bearer token never reaches the browser. A small `alert-snapshot` Stimulus controller hides the figure when no snapshot exists (older/evicted alerts), avoiding broken-image icons.
- Added an operator-only BearClaw Runs viewer under `/agent/runs` with a recent-runs index, run detail timeline, and a browser SSE consumer that replays/tails live tool-call events through the Rails app without exposing BearClaw bearer headers to the browser.
- Added graceful unavailable states for the Runs pages when BearClaw cannot be reached, plus regression coverage for the login → list runs → open run flow and HTML redaction of rendered run artifacts.
- Added an operator-only Tardigrade transcript viewer under `/agent/transcripts` with recent redacted transcript history, transcript detail pages, and regression coverage for the login → list transcripts → open transcript flow.

### Fixed

- Fixed widget picker label text invisible in system tests. The `lg:grid-cols-[minmax(0,1.4fr)_minmax(0,1fr)]` two-column grid in `_widget_picker.html.erb` left only 124 px for the capability list column at a 1400 px viewport. After nested padding the `<div class="min-w-0">` containing the capability-name `<p>` elements collapsed to zero width, causing Chrome's `innerText` (used by Capybara `element.text`) to return an empty string for the truncated text. Removed the responsive two-column grid so the full aside width is available.
- Fixed `DashboardLayoutPresetsTest` asserting `"Size 2×2"` after applying a saved preset. The `DashboardDensityUpgrader` runs on the first page visit and upgrades legacy 8-column dashboards (×10 scale factor), so a 2×2 tile becomes 20×20 before the preset is saved. The test now correctly asserts `"Size 20×20"`.

- Fixed `_dashboard_editor.html.erb` NameError: Service Providers / Connections / Devices sections used bare local names (`service_providers`, `service_connections`, `devices`) that were never passed as locals. Wrapped those three admin-only sections in `<% if current_user.admin? %>` (preserving the `assert_no_match "Service Providers"` assertion for operator users) and changed the references to the `@` instance variables the controller already sets.

- Fixed system test null-reference JS crash in `dashboard_layout_test.rb`. The grid element has `data-controller="square-grid dashboard-layout"` (two Stimulus controllers sharing one element), but the test helpers used `[data-controller='dashboard-layout']` (CSS exact-match), which never matched. Changed to `[data-controller~='dashboard-layout']` (CSS word-list match) in the Capybara `assert_selector` and both `document.querySelector` calls inside `move_tile`/`resize_tile` so the grid element is correctly found.

- Fixed recurring bearclaw-web→Koala MCP `401`s on deploy. `KOALA_TOKEN` was listed in the homelab Blink provisioner's `always_update`, so every deploy overwrote the correct token with a stale placeholder coming from the Blink MCP server's process environment (which overrides the `.env` files). Removed `KOALA_TOKEN` from `always_update` so it is seeded once and preserved across deploys, consistent with `URSA_TOKEN`/`KODIAK_TOKEN`/`POLAR_TOKEN`; a clean redeploy now preserves the token and passes the full Blink verify suite (14/14) with no manual intervention.
- Fixed test env-var leakage that failed the deploy verify gate. `LoginPageTest` and `Settings::XIntegrationsControllerTest` depended on `OIDC_SUPPORT_ENABLED` / `X_REDIRECT_URI` being unset, but the homelab/container environment sets them — so the support-login button and a configured X callback URL leaked into the tests. The tests now isolate those env vars in setup/teardown (feature code unchanged). Also corrected `Home::DashboardResetterTest`'s stale tile-geometry expectation (`[1,1,2,2]`→`[1,1,20,20]`) to match the 80-column density rework (restore seeds a 1×1 tile, the density upgrader scales ×20).

- Fixed the BearClawWeb Blink env seed for support OIDC login. The manifest was using shell-style default syntax (`${OIDC_SUPPORT_ENABLED:-false}`), but Blink treats the whole token as a variable name instead of applying shell expansion. Provisioning now reads the real `OIDC_SUPPORT_ENABLED` value from the deploy environment or root `.env` file.
- Fixed inconsistent Home dashboard grid sizing — and the edit/view mismatch where a camera tile rendered shorter in edit mode than view mode — by making the square-cell behavior a single always-on `square-grid` Stimulus controller. It pins `grid-auto-rows` to the measured column-track width (via `ResizeObserver`) in both view and edit modes, so a tile's pixel width:height matches its grid-unit span and proportions stay consistent. The edit-only `dashboard-layout` controller no longer owns row-track sizing or re-fits camera tiles, removing the conflict that produced two different heights for the same tile.
- Made camera-tile height correction deterministic and server-side. A one-time `Home::CameraAspectBackfill` (guarded by a settings version) snaps each existing camera tile's height to `round(width / feed_aspect)` so the tile box hugs the 16:9 feed, replacing the earlier JS viewport-derived persist that depended on the live container width at load. New/quick-added camera tiles get the same feed-shaped height from `Dashboard#default_camera_tile_height`. Drag-resize stays aspect-locked: because cells are square, the dependent dimension is simply `round(width / feedAspect)` (or its inverse) from the drag's dominant axis.

### Changed

- Fixed distorted (horizontally stretched) camera feeds and made camera tiles hug the feed. The live frame now renders with `object-contain`, so the source aspect ratio is always preserved — the feed can never be squished or stretched. The actual stretch was caused by the `object-contain` utility being absent from the compiled Tailwind build (Tailwind only compiles classes it has seen, and only `object-cover` had ever been used); `assets:precompile`/`tailwindcss:build` now regenerates it. Camera-tile resizing is now aspect-locked: the camera feed publishes its real `naturalWidth/naturalHeight` to the tile (seeded server-side at 16:9), and the dashboard resize handle derives the dependent dimension from the drag's dominant axis, so dragging a camera tile keeps the feed's proportions and the tile box hugs the feed instead of letterboxing. The camera frame's `min-height` floor was lowered (12rem → 8rem) so the grid height actually governs the tile. (See the consolidated `square-grid` controller and `Home::CameraAspectBackfill` notes under "Fixed" for the final, deterministic implementation that supersedes the earlier viewport-derived correction.)
- Reworked Home dashboard editing and camera layout density around a much finer `80×60` grid. Existing dashboards are now upgraded from the coarse dense layout into 10x finer placement units, so drag/resize interactions feel substantially less rigid and camera tiles can be tuned in much smaller increments.
- Camera-feed widgets no longer force a square viewport. Single-camera tiles now let the feed fill the tile body, which removes most of the wasted letterboxed space and makes rectangular camera tiles usable again.
- Full-bleed camera tiles no longer grow a separate header/footer in edit mode. The live feed keeps the exact same size and shape in both modes, and edit/delete controls now overlay the camera image instead of stealing layout space from the widget.
- Polar air-quality stat tiles now use the same chrome-free edit treatment: no tile header/footer around single-widget metric cards, identical view/edit sizing, and edit/delete controls overlaid directly on the widget instead of wrapped in extra dashboard chrome.
- Recent Alerts snapshot images are now clickable. Clicking a detection frame opens a centered larger modal preview so operators can inspect people/package captures without leaving the dashboard.
- Replaced the Home dashboard edit sidebar with a compact quick-add panel and tile-corner edit/delete controls. Edit mode now focuses on searching a capability and adding it to the dashboard, while tile/widget settings move into per-tile modals instead of one extremely long repeated form stack.
- Dashboard tile resize now uses the active tile's own grid geometry and locks the corner drag to width or height when one axis clearly dominates. That makes it practical to widen a tile into a rectangle or make it taller without the resize handle feeling square-locked.
- Camera-feed widgets now keep their square viewport without cropping the scene: frames render with `object-contain` inside the square instead of `object-cover`, and legacy 4-second camera refresh intervals are automatically nudged to 2 seconds so the home camera wall feels more live.
- Camera-feed widgets now render inside a square viewport instead of a shallow wide strip, so the live Koala frames crop cleanly without the stretched letterbox look on the home dashboard.
- BearClawWeb now emits its own Content Security Policy with importmap script nonces, websocket-friendly `connect-src`, permissive `img-src` for same-origin snapshots and HTTPS avatars, and `style-src 'unsafe-inline'` for the dashboard's existing inline grid styles. This prevents Tardigrade from injecting the fallback `default-src 'self'` policy that was blocking Stimulus startup, leaving camera feeds black, and freezing the dashboard sync indicator at `Connecting…`.
- Tightened the Home dashboard's "Recent Alerts" and "Priority Watch" sections with denser card spacing, slimmer headers, and four-up large-screen packing. Both sections now include a persistent collapse/expand control so operators can reclaim vertical space without losing quick access to active alerts.
- Camera-feed widgets are now health-checked by frame capture, not scene activity: a `camera_feed` widget always reports `:healthy` unless its capability status is `unavailable`/`degraded` (a real capture failure). A quiet scene with an old frame timestamp no longer marks the tile stale; detection events are surfaced via the alerts feed instead.
- Mounted the Rails ActionCable endpoint at `/cable`, emitted the `action-cable-url` meta tag in the shared layout, and made `SyncDashboardJob` resolve the household owner's home dashboard instead of assuming the active viewer owns one. Home dashboard viewers now trigger an immediate sync on websocket connect, so Polar-backed air-quality widgets refresh for shared-household members instead of sitting on stale cached readings.
- Fixed the homelab BearClawWeb Blink provisioner so it preserves the real `KOALA_TOKEN`, collapses duplicate runtime env keys before container recreation, and verifies the current bearer-only `403 asserted tardigrade identity required` behavior at the public BearClaw edge.
- Updated the homelab BearClawWeb deploy contract to follow Koala's current public port (`6705`), expanded the env sync so `POLAR_URL`, `KODIAK_URL`, and `URSA_URL` are rewritten alongside `KOALA_URL`, and stopped the remote provisioning script from overwriting the Koala MCP token with a hardcoded dev value. The public BearClaw chat verify step now expects the current `403 asserted tardigrade identity required` response for bearer-only calls through the edge.
- Updated `KodiakClient` to use the versioned `/api/v1/` path prefix, unwrap the Kodiak v1 response envelope (`{"data": ..., "error": null, "meta": {...}}`), extract human-readable messages from error envelopes, and propagate `X-BearClaw-Actor` / `X-BearClaw-Role` headers from `current_user` on every request. `strategies` now returns the inner array directly so views receive the expected shape. `base_controller.rb` passes `current_user.email` and role to the client constructor.
- BearClaw operator login now issues a scoped HS256 Tardigrade identity token into the Rails session, and BearClaw client calls now preserve configured edge base paths so `BEARCLAW_URL` can target the `/bearclaw` mount instead of talking straight to the private loopback port.
- Moved BearClawWeb host port from `3001` to `6701`; updated `config/environments/production.rb` `config.hosts` to allow `127.0.0.1:6701` to align with the homelab port scheme. Updated `blink.toml` (target + service port, port-published verify regex), `deploy/blink/provision_bearclaw_web.sh` (PORT variable), `BLINK.md`, and `CLAUDE.md`. Tardigrade's `blink.toml` verify check updated to match.
- Standardized the repository documentation contract and documented the project-local Blink deployment in `BLINK.md`.
- Stopped pinning the production BearClaw container to `192.168.86.53` for DNS and added a Blink verify check for Google OAuth DNS resolution from inside the container.
- Ignored the repository-root `blink.toml` and `BLINK.md` and stopped tracking them so homelab-specific Blink targets and operator notes stay local-only.
